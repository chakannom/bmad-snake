# Product Requirements Document - bmad-snake

**Author:** Kai
**Date:** 2026-04-12

## Executive Summary

`bmad-snake`는 브라우저에서 즉시 실행되는 반응형 캐주얼 Snake 게임으로, 짧은 플레이 시간 안에 스트레스를 해소하는 경험을 제공하는 것을 목표로 한다. 주요 타깃은 PC와 모바일 사용자이며, 별도 설치/가입 없이 바로 시작해 2~5분 단위로 가볍게 즐길 수 있도록 설계한다.  
본 프로젝트는 학습용 데모 성격의 greenfield 프로젝트로, BMAD 방법론을 적용해 기획-구현-검증 사이클을 짧게 반복하며 핵심 재미 루프를 검증한다. MVP 성공 기준은 평균 플레이 시간 3분 이상 달성이며, 이를 위해 스테이지 완료율과 재도전률을 함께 관찰한다.

### What Makes This Special

이 제품의 차별점은 고전 Snake를 무한 생존형이 아닌 **시간 제한 기반 스테이지 공략형 구조**로 재해석한 점이다. 사용자는 20개 맵을 4개 티어 난이도로 순차 공략하며, “짧은 집중-즉시 재도전-진행 성취” 루프를 반복한다.  
핵심 인사이트는 복잡한 성장/소셜 요소 없이도, 명확한 제한 시간과 목표 피드백만으로 높은 몰입과 스트레스 해소 가치를 만들 수 있다는 점이다. MVP에서는 랭킹 등 경쟁 요소를 의도적으로 제외해 코어 게임플레이 품질과 모바일 조작 완성도에 집중한다.

## Success Criteria

### User Success
- 사용자는 첫 진입 후 10초 이내 게임을 시작할 수 있다(설치/가입 없음).
- 사용자는 평균 3분 이상 플레이하며, “짧은 시간 스트레스 해소” 목적을 충족한다고 느낀다.
- 초기 구간 난이도는 학습 가능한 수준을 유지해 스테이지 1 완료율이 안정적으로 확보된다.
- 실패 후 재도전 루프가 빠르게 동작해 즉시 재시작률이 높게 유지된다.

### Business Success
- 학습용 데모 목표 달성: BMAD 방법론 기반 기획-구현-검증 흐름이 문서와 결과물로 재현 가능해야 한다.
- 제품 가치 검증: 평균 플레이 시간 3분 이상을 달성해 핵심 루프(짧은 몰입+해소)가 유효함을 입증한다.
- 플랫폼 적합성 검증: PC/모바일 모두에서 핵심 플레이가 동일한 품질로 동작한다.

### Technical Success
- 주요 브라우저 환경에서 반응형 UI 및 입력(키보드/터치 스와이프)이 안정 동작한다.
- 스테이지 진행 로직(제한 시간, 목표 달성, 실패 조건, 다음 맵 해금)이 결함 없이 일관 동작한다.
- 모바일에서 HUD 가독성과 입력 정확도가 플레이를 방해하지 않는 수준을 달성한다.
- 세션/이벤트 계측(시작, 종료, 플레이 시간, 스테이지 도달/완료, 재시작)이 정확히 수집된다.

### Measurable Outcomes
- Primary KPI: 평균 플레이 시간 >= 3분/세션.
- 관찰 지표:
  - 스테이지 1 완료율
  - 스테이지 5 도달률
  - 실패 후 즉시 재시작률
- 측정 정의:
  - 세션 시작: 첫 입력 시점
  - 세션 종료: 게임오버/클리어 후 종료/30초 비활성

## Product Scope

### MVP - Minimum Viable Product
- 브라우저 기반 싱글플레이 Snake
- PC + 모바일 반응형 지원
- 시간 제한 스테이지 모드(20맵, 4티어 난이도)
- 클리어/실패/재시작/해금 루프
- KPI 측정을 위한 기본 텔레메트리

### Growth Features (Post-MVP)
- 고스트 리플레이(우선순위 1)
- 일일 챌린지 모드
- 테마/환경 변형 맵 팩

### Vision (Future)
- 짧은 세션 기반 스트레스 해소 웹게임으로 포지셔닝 고도화
- 개인 퍼포먼스 개선 경험(고스트 비교)을 중심으로 재방문 강화
- 필요 시 경쟁/소셜 요소(예: 랭킹)는 코어 루프 검증 이후 단계적으로 검토

## User Journeys

### 1) Primary User - Success Path (PC 사용자: 민수)
- Opening Scene: 민수는 업무 중 10분 휴식 시간에 머리를 비우고 싶지만, 복잡한 게임은 부담스럽다.
- Rising Action:
  - 검색/북마크로 게임 페이지 진입
  - 즉시 시작 버튼으로 1스테이지 진입
  - 제한 시간과 목표 먹이 수를 HUD로 확인
  - 키보드 조작으로 장애물을 피해 목표 달성
- Climax: 제한 시간 3초 전 마지막 먹이를 먹고 스테이지 클리어, 다음 맵 해금.
- Resolution: “짧게 했는데 꽤 개운하다”는 감각을 얻고, 휴식마다 재방문한다.
- 실패/복구 포인트:
  - 조작 실수로 충돌 -> 즉시 재시작
  - 시간 초과 -> 목표치/경로를 보고 재도전

### 2) Primary User - Edge Case (모바일 사용자: 지은)
- Opening Scene: 지은은 지하철에서 한 손으로 게임하려고 한다.
- Rising Action:
  - 모바일 브라우저 진입 후 세로 화면에서 플레이 시작
  - 스와이프 입력으로 방향 전환
  - 중간에 알림/앱 전환으로 잠시 이탈
- Climax: 복귀 후 남은 시간 압박 속에서 경로를 바꿔 겨우 클리어 또는 실패.
- Resolution: 실패해도 즉시 재도전 가능해 이탈 없이 2~3판 연속 플레이.
- 실패/복구 포인트:
  - 스와이프 오인식 -> 입력 버퍼/민감도 최적화 필요
  - 작은 화면 HUD 가독성 저하 -> 핵심 정보 우선 노출 필요

### 3) Admin/Operations User (운영자: Kai)
- Opening Scene: Kai는 학습용 데모의 품질을 검증하고 BMAD 결과물을 정리해야 한다.
- Rising Action:
  - 버전별 맵 난이도/타이머를 조정
  - 배포 후 텔레메트리(평균 플레이 시간, 스테이지 완료율, 재시작률) 확인
  - PC/모바일 지표를 분리 비교
- Climax: 평균 플레이 시간 3분 미달 구간을 발견하고 초반 맵 난이도/시간을 조정.
- Resolution: 코어 루프가 안정되며 데모 목표(학습+검증)를 충족.
- 실패/복구 포인트:
  - 데이터 누락 -> 이벤트 정의 재검토 및 로그 검증
  - 특정 맵 이탈 급증 -> 맵 단위 밸런스 패치

### 4) Support/Troubleshooting User (테스터/리뷰어: Alex)
- Opening Scene: Alex는 “모바일에서 조작이 답답하다”는 피드백을 수집한다.
- Rising Action:
  - 재현 시나리오(기기/브라우저/화면방향)로 문제를 재현
  - 입력 이벤트와 세션 종료 이벤트를 대조
  - 문제를 “입력 인식 지연” vs “난이도 과도”로 분류
- Climax: 특정 기기에서 스와이프 임계값이 높아 오인식이 발생함을 확인.
- Resolution: 민감도 조정안/재테스트 체크리스트를 운영자에게 전달.
- 실패/복구 포인트:
  - 재현 불가 -> 최소 재현 조건 템플릿 필요
  - 로그 부족 -> 디버그 이벤트 추가 필요

### 5) API/Integration (현재 비적용)
- 현재 MVP는 외부 API 소비자 여정이 없다.
- 단, V2 이후 리더보드/소셜 연동을 고려하면 API 소비자(내부/외부) 여정을 별도 정의할 수 있다.

### Journey Requirements Summary
- 즉시 시작 UX: 설치/가입 없는 진입과 빠른 재시작 플로우
- 입력/조작 안정성: 키보드/스와이프 정확도, 모바일 가독성
- 스테이지 시스템: 시간 제한, 목표 달성, 실패 처리, 해금 루프
- 관측 가능성: 세션/스테이지/재시작 이벤트 계측 및 플랫폼별 분석
- 운영 가능성: 맵 난이도·시간 튜닝과 품질 피드백 반영 루프
- 지원 가능성: 문제 재현 템플릿과 디버깅 로그 체계

## Domain-Specific Requirements

### Compliance & Regulatory
- 본 MVP는 비규제 캐주얼 게임으로, 의료/금융 수준의 강한 규제 요건은 없음.
- 다만 웹 서비스 기본 준수 항목은 포함:
  - 개인정보 최소 수집 원칙(익명 플레이 우선)
  - 분석 이벤트 수집 시 개인정보 비식별 처리
  - 필요 시 쿠키/로컬스토리지 고지

### Technical Constraints
- 입력 반응성:
  - 키보드/터치 입력 지연이 체감되지 않도록 프레임 안정성 확보
- 크로스 디바이스 일관성:
  - PC/모바일에서 동일 룰로 플레이 가능해야 함
- 접근성 최소 기준:
  - HUD 대비(가독성), 색상 의존 최소화, 핵심 상태(시간 임박/실패) 명확 표시
- 데이터 보존:
  - MVP는 로컬 저장 중심(간단한 진행 상태/설정), 서버 영속 저장은 후순위

### Integration Requirements
- MVP 필수 외부 연동 없음
- 선택적 연동(후속):
  - 웹 분석 도구(세션/리텐션/스테이지 진행)
  - 에러 로깅 도구(모바일 입력 이슈 진단)

### Risk Mitigations
- 리스크: 모바일 입력 오인식
  - 대응: 스와이프 임계값 튜닝 + 입력 버퍼 + 기기별 테스트 시나리오
- 리스크: 초기 난이도 과도
  - 대응: 1~5맵 클리어율 기준으로 타이머/맵 밀도 조정
- 리스크: 성능 저하로 몰입 붕괴
  - 대응: 저사양 기기 기준 성능 점검, 렌더링 비용 상한 관리
- 리스크: KPI 해석 불가(이벤트 정의 불일치)
  - 대응: 세션 시작/종료/재시작 이벤트 스키마 고정 및 검증

## Innovation & Novel Patterns

### Detected Innovation Areas
- 고전 Snake를 무한 생존형에서 **시간 제한 스테이지 공략형**으로 전환해, “짧은 시간 스트레스 해소”라는 목적 중심 경험으로 재설계.
- 20맵/4티어 구조를 통해 “짧은 집중-실패-즉시 재도전” 루프를 명확히 제품화.
- 학습용 데모 맥락에서 KPI(평균 플레이 시간) 중심으로 게임 밸런스를 운영하는 **데이터 피드백 루프**를 명시.

### Market Context & Competitive Landscape
- 캐주얼 웹게임 다수는 광고/성장 메타 중심으로 설계되어 즉시 몰입 저해 요소가 존재.
- 본 제품은 복잡도보다 “짧은 세션 완결성”에 집중해 대체재와 차별화.
- 강한 기술 혁신보다는 UX 루프 혁신(세션 구조, 목표 피드백, 재도전 템포)에 무게를 둠.

### Validation Approach
- 핵심 가설: 시간 제한 스테이지 구조가 평균 플레이 시간 3분 이상을 달성하게 만든다.
- 검증 방법:
  - 스테이지 1 완료율/스테이지 5 도달률/즉시 재시작률 추적
  - 맵 난이도·제한시간 A/B 튜닝으로 체류/이탈 변화 관찰
  - PC/모바일 분리 분석으로 플랫폼별 UX 병목 확인

### Risk Mitigation
- 리스크: “혁신”이 사용자에게는 단순한 난이도 상승으로만 인식될 수 있음  
  대응: 목표/보상 피드백을 명확히 보여 체감 가치를 강화
- 리스크: 초반 피로도 증가로 빠른 이탈  
  대응: 1티어 난이도 완화, 재시작 마찰 최소화
- 리스크: 데이터는 수집되나 의사결정에 연결되지 않음  
  대응: KPI 임계치 기반 밸런스 조정 규칙 사전 정의

## Web App Specific Requirements

### Project-Type Overview
- 제품은 설치 없이 즉시 접속 가능한 브라우저 게임으로, 짧은 세션 플레이와 반응형 UX가 핵심이다.
- 기술 우선순위는 기능 폭보다 입력 반응성, 화면 가독성, 크로스 디바이스 일관성이다.

### Technical Architecture Considerations
- 아키텍처 형태: `SPA` 우선 (단일 게임 루프 중심, 빠른 상태 전환).
- 실시간 통신: MVP에서는 불필요 (`no real-time backend required`).
- 상태 관리: 게임 상태(맵, 타이머, 스코어, 진행도)는 클라이언트 중심 관리.
- 데이터 전략: MVP는 로컬 저장 우선, 서버 동기화는 후속.

### Browser Matrix
- 지원 대상(초기):
  - Desktop: 최신 Chrome, Safari, Edge (최근 2개 메이저 버전)
  - Mobile: iOS Safari, Android Chrome (최근 2개 메이저 버전)
- 비지원/후순위:
  - 구형 브라우저 및 레거시 엔진 기반 환경

### Responsive Design Requirements
- 모바일/PC 공통 게임 룰 유지, 레이아웃만 적응형으로 변경
- HUD(남은 시간, 목표, 길이)는 화면 크기에 따라 우선순위 기반 배치
- 입력 UI:
  - PC: 키보드 화살표/WSAD
  - Mobile: 스와이프 중심 + 오입력 완화 버퍼

### Performance Targets
- 플레이 중 프레임 드랍이 체감되지 않는 수준 유지
- 입력 반응은 즉각적으로 인지 가능한 수준으로 유지
- 저사양 모바일에서도 코어 루프(이동/충돌/타이머)가 끊기지 않아야 함

### SEO Strategy
- 핵심 플레이 페이지는 SEO보다 즉시 플레이 경험 우선
- 필요 시 랜딩 섹션(소개/가치/플레이 유도)만 최소 메타 최적화
- 학습용 데모 성격상 검색 유입은 1차 KPI가 아님

### Accessibility Level
- 목표 수준: 실사용 최소 접근성 기준 충족 (핵심 화면 가독성 중심)
- 필수 항목:
  - 텍스트/배경 대비 확보
  - 색상만으로 상태 전달하지 않기
  - 시간 임박/실패 상태의 명확한 시각 신호
- 확장 항목(후속):
  - 추가 입력 옵션, 모션/시각 보조 설정

### Implementation Considerations
- 웹앱 특성상 배포 단순성(정적 호스팅 가능)과 빠른 반복 개선이 중요
- 맵 난이도 조정과 KPI 분석을 빠르게 반복할 수 있도록 이벤트 스키마를 먼저 고정
- `skip_sections` 적용:
  - native_features, cli_commands 관련 요구사항은 PRD 범위에서 제외

## Project Scoping & Phased Development

### MVP Strategy & Philosophy
- MVP Approach: **Experience MVP + Validation MVP**
- 전략:
  - “짧은 시간 스트레스 해소” 경험을 가장 작게 완성
  - KPI(평균 플레이 시간 3분)로 가치 가설 검증
- Resource Requirements:
  - 최소 1인(기획+개발+테스트 겸임)로 가능
  - 권장 2인(개발 1, QA/밸런싱 1) 시 반복 속도 개선

### MVP Feature Set (Phase 1)

**Core User Journeys Supported:**
- PC 사용자의 빠른 진입-플레이-재도전 루프
- 모바일 사용자의 스와이프 기반 플레이 및 복구 루프
- 운영자 지표 확인 및 난이도/시간 튜닝 루프

**Must-Have Capabilities:**
- 브라우저 즉시 실행(설치/가입 없음)
- 싱글플레이 시간 제한 스테이지 모드
- 20맵(4티어) 진행/해금/재시작
- PC 키보드 + 모바일 스와이프 입력
- 핵심 HUD(시간/목표/상태) 가시성 확보
- 이벤트 계측(세션/도달/완료/재시작)

### Post-MVP Features

**Phase 2 (Post-MVP):**
- 고스트 리플레이(우선순위 1)
- 일일 챌린지
- 난이도 자동 튜닝 실험

**Phase 3 (Expansion):**
- 테마 맵 팩
- 선택형 경쟁 요소(리더보드 등) 재검토
- 외부 공유/소셜 확장(가치 검증 후)

### Risk Mitigation Strategy
- Technical Risks:
  - 모바일 입력 오인식, 저사양 성능 저하
  - 대응: 입력 임계값/버퍼 조정, 성능 예산 관리
- Market Risks:
  - “재미는 있으나 금방 질림” 위험
  - 대응: 초반 난이도 곡선 최적화, 재도전 보상 피드백 강화
- Resource Risks:
  - 1인 개발 시 일정 지연 가능
  - 대응: 맵/기능 고정 후 밸런싱 범위 제한, Phase 2 기능 동결

## Functional Requirements

### Game Session & Access
- FR1: 사용자는 브라우저에서 설치/가입 없이 게임을 시작할 수 있다.
- FR2: 사용자는 메인 진입점에서 즉시 새 세션을 시작할 수 있다.
- FR3: 시스템은 사용자 첫 입력 시 세션 시작을 기록할 수 있다.
- FR4: 시스템은 게임오버, 클리어 종료, 비활성 종료를 구분해 세션 종료를 기록할 수 있다.

### Core Gameplay
- FR5: 사용자는 Snake를 방향 입력으로 조작할 수 있다.
- FR6: 시스템은 뱀 이동, 충돌, 먹이 획득 규칙을 일관되게 적용할 수 있다.
- FR7: 시스템은 스테이지별 제한 시간을 적용할 수 있다.
- FR8: 시스템은 스테이지별 목표 먹이 수를 적용할 수 있다.
- FR9: 시스템은 제한 시간 내 목표 달성 시 스테이지 클리어를 처리할 수 있다.
- FR10: 시스템은 충돌 또는 시간 초과 시 실패를 처리할 수 있다.
- FR11: 사용자는 실패 후 즉시 재시작할 수 있다.

### Stage Progression
- FR12: 시스템은 총 20개 스테이지를 제공할 수 있다.
- FR13: 시스템은 스테이지를 4개 티어 난이도 구조로 관리할 수 있다.
- FR14: 시스템은 클리어 결과에 따라 다음 스테이지를 해금할 수 있다.
- FR15: 사용자는 현재 도달 가능한 스테이지를 확인하고 선택할 수 있다.
- FR16: 시스템은 스테이지별 진행 상태를 유지할 수 있다.

### Cross-Platform Interaction
- FR17: 사용자는 PC 환경에서 키보드 입력으로 플레이할 수 있다.
- FR18: 사용자는 모바일 환경에서 스와이프 입력으로 플레이할 수 있다.
- FR19: 시스템은 입력 오인식 완화를 위한 입력 처리 정책을 적용할 수 있다.
- FR20: 시스템은 모바일/PC에서 동일한 핵심 게임 규칙을 제공할 수 있다.

### Gameplay Feedback & HUD
- FR21: 시스템은 플레이 중 남은 시간 정보를 표시할 수 있다.
- FR22: 시스템은 플레이 중 목표 진행 상태를 표시할 수 있다.
- FR23: 시스템은 플레이 중 핵심 상태 변화를 명확히 전달할 수 있다.
- FR24: 시스템은 스테이지 클리어 및 실패 결과를 사용자에게 명확히 전달할 수 있다.
- FR25: 시스템은 작은 화면에서도 핵심 플레이 정보를 우선 표시할 수 있다.

### Telemetry & Analytics
- FR26: 시스템은 세션별 플레이 시간을 수집할 수 있다.
- FR27: 시스템은 스테이지 진입/완료/실패 이벤트를 수집할 수 있다.
- FR28: 시스템은 실패 후 재시작 이벤트를 수집할 수 있다.
- FR29: 시스템은 플랫폼(PC/모바일) 기준으로 지표를 분리 집계할 수 있다.
- FR30: 시스템은 평균 플레이 시간을 산출할 수 있다.

### Operations & Tuning
- FR31: 운영자는 스테이지 난이도 파라미터를 조정할 수 있다.
- FR32: 운영자는 스테이지 제한 시간을 조정할 수 있다.
- FR33: 운영자는 지표를 기반으로 밸런스 조정 결정을 내릴 수 있다.
- FR34: 시스템은 운영자가 문제 상황을 진단할 수 있도록 플레이/이벤트 기록을 제공할 수 있다.

### Data & Privacy Baseline
- FR35: 시스템은 MVP 범위에서 로컬 기반으로 사용자 진행 상태를 저장할 수 있다.
- FR36: 시스템은 분석 이벤트 수집 시 개인정보를 최소화할 수 있다.
- FR37: 시스템은 필요 시 데이터 저장/수집 고지 정보를 제공할 수 있다.

### Post-MVP Capability Hooks
- FR38: 시스템은 향후 고스트 리플레이 기능이 추가될 수 있도록 플레이 기록 확장 여지를 둘 수 있다.
- FR39: 시스템은 향후 일일 챌린지 규칙을 적용할 수 있도록 스테이지 규칙 확장 여지를 둘 수 있다.
- FR40: 시스템은 향후 테마 맵 확장을 수용할 수 있다.

## Non-Functional Requirements

### Performance
- NFR1: 게임 플레이 중 사용자 입력부터 방향 반영까지의 지연은 사용자가 즉시 반응으로 인지할 수 있는 수준이어야 한다.
- NFR2: 일반 플레이 상황에서 게임 루프는 시각적으로 끊김 없는 상태를 유지해야 한다.
- NFR3: 스테이지 시작, 재시작, 결과 전환은 플레이 흐름을 방해하지 않는 짧은 대기 시간 내에 완료되어야 한다.
- NFR4: 저사양 모바일 환경에서도 코어 루프(이동, 충돌, 타이머)는 기능 저하 없이 동작해야 한다.

### Reliability
- NFR5: 세션 시작/종료, 스테이지 진입/완료/실패, 재시작 이벤트는 누락 없이 기록되어야 한다.
- NFR6: 브라우저 탭 전환 또는 일시 중단 이후 복귀 시 게임 상태는 정의된 정책(재개/종료)에 따라 일관되게 처리되어야 한다.
- NFR7: 예기치 않은 오류 발생 시 사용자는 현재 세션 실패를 인지하고 즉시 재시작할 수 있어야 한다.

### Security & Privacy
- NFR8: MVP에서 수집하는 데이터는 플레이 분석에 필요한 최소 범위로 제한되어야 한다.
- NFR9: 분석 이벤트에는 개인 식별이 가능한 정보가 포함되지 않아야 한다.
- NFR10: 로컬 저장 데이터는 게임 진행에 필요한 범위로 제한되어야 하며, 민감정보를 저장하지 않아야 한다.

### Accessibility
- NFR11: HUD의 핵심 텍스트/상태 정보는 다양한 화면 크기에서 읽을 수 있는 대비와 크기를 유지해야 한다.
- NFR12: 시간 임박, 실패, 클리어 상태는 색상 외 추가 시각 신호로 구분되어야 한다.
- NFR13: 주요 플레이 기능은 터치/키보드 각각에서 단일 입력 방식으로 수행 가능해야 한다.

### Compatibility
- NFR14: MVP는 최신 주요 브라우저(Desktop: Chrome/Safari/Edge, Mobile: iOS Safari/Android Chrome 최근 2개 메이저 버전)에서 동일한 핵심 게임 규칙을 제공해야 한다.
- NFR15: 지원 대상 브라우저에서 레이아웃 깨짐 없이 플레이 가능한 반응형 화면을 제공해야 한다.
